### 3.2 Findings

**âš ď¸Ź FINDING â€” AudioManager writes localStorage during active gameplay**  
**File:** `games/missile-command-math/src/systems/AudioManager.ts`, **line 288**

```typescript
// In setEnabled(enabled: boolean):
try {
  localStorage.setItem(SOUND_ENABLED_KEY, enabled ? '1' : '0');
} catch {
  // localStorage not available
}
```

`AudioManager.setEnabled()` is called from `wireEventListeners()` in response to the `SOUND_TOGGLED` event, which fires when the player toggles sound from the in-game pause menu during a live wave. This is a `localStorage` write during gameplay, which is **prohibited** by CLAUDE.md:

> *"No localStorage during gameplay. In-memory only."*

The `SettingsScene.onSoundToggle()` (line 182) correctly persists the sound preference when accessed from the main Settings menu (outside gameplay), so persistence is already covered. The write in `AudioManager.setEnabled()` is redundant and a policy violation.

**Recommendation:** Remove the `localStorage.setItem` call from `AudioManager.setEnabled()`. The method should update only the in-memory `this.soundEnabled` flag and the Web Audio gain nodes. Persistence is the responsibility of the Settings scene, not the real-time audio engine.

---

**âš ď¸Ź FINDING â€” `mcm_stars` and `mcm_highest_level` are never written back to localStorage**  
These keys are read in `BootScene.ts` (lines 123â€“140) and loaded into the Phaser registry, but no code path ever writes updated star ratings or the highest-unlocked-level back to `localStorage`. The `LevelCompleteScene` updates are in-memory only (via Phaser registry). Progress is therefore **not persisted across browser sessions**.

While this is not a *security* violation, it is noted because:
1. The `LevelSelectScene.ts` JSDoc (lines 9â€“10) documents these keys as if they are a reliable persistence store, implying a write path was intended.
2. The `shared/analytics` module defines `STORAGE_KEYS.LEVEL_STARS = 'arcade-swarm:mcm:level-stars'` (a different key than `mcm_stars`) suggesting a write path may have been planned but not yet implemented.

**Recommendation:** Assign this finding to the Gameplay or Engine agent for follow-up. The write should occur in `LevelCompleteScene` (outside of active gameplay), making it policy-compliant.

---

**âš ď¸Ź FINDING â€” Inconsistent localStorage key namespace**  
`shared/analytics/src/index.ts` defines `STORAGE_KEYS` with the prefix `arcade-swarm:mcm:` (e.g., `arcade-swarm:mcm:high-scores`), but the game scenes use the `mcm_` prefix (`mcm_sound_enabled`, `mcm_difficulty`, `mcm_stars`, `mcm_highest_level`, `mcm_has_seen_training`). The analytics module's constants are **never used** by any game scene â€” the scenes define their own inline key strings.

This creates two separate namespaces with no cross-reference, risking future confusion about which keys are canonical. It is not a security issue today, but a hygiene concern worth flagging.

**Recommendation:** Consolidate all `localStorage` key constants into a single source-of-truth module (e.g., extend `shared/analytics/src/index.ts` `STORAGE_KEYS` to cover all keys, or export a `STORAGE_KEYS` from `games/.../src/config/`). Remove the duplicate key constants scattered across scene files.

A **live Anthropic API key** is committed in the `.env` file at the repository root. This is a critical secret exposure.

**Mitigating factor:** `.gitignore` correctly lists `.env` and `.env.*` (excluding `.env.example`), so this key is not tracked by git and **will not be pushed to remote** under normal git operations. However:

- If the repository has already been pushed with this file accidentally (e.g., before `.gitignore` was in place), the key may already be in git history.
- The key is exposed to any local process that reads the `.env` file, including any tool run in this repository.
- If the repository is shared as a zip, tarball, or direct copy, the `.env` file will be included.

**Required actions (in order of urgency):**
1. **Immediately rotate the key** at [console.anthropic.com](https://console.anthropic.com) â€” treat this key as compromised.
2. Run `git log --all --full-history -- .env` to confirm the file is not tracked in git history. If it is, rewrite history with `git filter-repo` or BFG Repo Cleaner.
3. Replace the value in `.env` with `sk-ant-your-key-here` (matching `.env.example`) or delete the local `.env` and re-add it from a password manager.
